# Architecture Design: LaCuisine

This document details the technical design for the LaCuisine application, focusing on the Next.js frontend, FastAPI backend, and Google Gemini AI integration.

## 1. System Architecture

The system follows a modern decoupled architecture:

*   **Frontend (Client):** A Next.js (React) application responsible for the UI/UX. It will communicate with the backend via RESTful APIs.
*   **Backend (API Provider):** A Python FastAPI application. It handles business logic, database interactions, and acts as the orchestrator for AI calls.
*   **Database:** PostgreSQL (hosted on Supabase or Neon).
*   **AI Service:** Google Gemini API (1.5 Flash).

### 1.1 Diagram

```mermaid
graph TD
    Client[Next.js Frontend] -->|REST API (JSON)| API[FastAPI Backend]
    API -->|SQL (asyncpg/SQLAlchemy)| DB[(PostgreSQL)]
    API -->|HTTPS Request (Images/Text)| LLM[Google Gemini API]
    LLM -->|JSON Response (Structured Recipe Data)| API
```

## 2. API Design (Backend - FastAPI)

The backend will expose endpoints for the frontend. Key endpoints include:

*   `POST /api/recipes/analyze-image`: Receives an image file (multipart/form-data), sends it to Gemini, and returns a structured JSON representing the recipe (`RecipeDraft`).
*   `POST /api/recipes`: Creates a new finalized recipe in the database.
*   `GET /api/recipes`: Lists recipes (with filtering/pagination).
*   `GET /api/recipes/{recipe_id}`: Retrieves a specific recipe with its full technical sheet.
*   `POST /api/menus/suggest`: Receives a base recipe (e.g., an appetizer) and returns suggested menus generated by Gemini based on the chef's database.
*   `POST /api/chat/copilot`: Interactive chat endpoint for generating recipe ideas from available ingredients.

## 3. Database Schema (PostgreSQL)

The core schema revolves around the `Recipe` and its components to form a professional technical sheet.

### Tables

*   `recipes`
    *   `id` (UUID, Primary Key)
    *   `title` (String)
    *   `category` (Enum: Appetizer, Main, Dessert, etc.)
    *   `style` (String)
    *   `yield_amount` (Decimal)
    *   `yield_unit` (String)
    *   `prep_time_minutes` (Integer)
    *   `temperature_type` (Enum: Hot, Cold)
    *   `image_url` (String)
    *   `created_at` (Timestamp)

*   `ingredients`
    *   `id` (UUID, Primary Key)
    *   `name` (String, Unique)

*   `recipe_ingredients`
    *   `id` (UUID, Primary Key)
    *   `recipe_id` (UUID, Foreign Key)
    *   `ingredient_id` (UUID, Foreign Key)
    *   `amount` (Decimal)
    *   `unit` (String)
    *   `notes` (String)

*   `recipe_steps`
    *   `id` (UUID, Primary Key)
    *   `recipe_id` (UUID, Foreign Key)
    *   `step_number` (Integer)
    *   `instruction` (Text)

## 4. AI Integration Details (Gemini)

### 4.1. Recipe Extraction Prompt (The "Digitador Universal")

When a user uploads a photo of a recipe, the backend will construct a prompt similar to this:

```text
System: You are an expert culinary assistant. Your task is to extract recipe information from the provided image and return ONLY a valid JSON object matching the requested schema. Do not include markdown formatting or explanations.

Schema:
{
  "title": string,
  "category": string (Appetizer, Main, Dessert, Other),
  "yield": { "amount": number, "unit": string },
  "ingredients": [
    { "name": string, "amount": number, "unit": string }
  ],
  "steps": [ string ]
}
```

### 4.2. Menu Suggestion Prompt (The "Sommelier")

```text
System: You are a professional chef specializing in menu design and flavor pairing. Look at the chef's available recipes provided below in JSON format.
User: I am serving {base_recipe_title} as the {category}. Plan a 3-course menu using ONLY the recipes provided in the context. Ensure the flavors balance well. Return the menu and a brief justification for the choices.
Context: {json_dump_of_available_recipes}
```

## 5. Deployment Strategy

*   **Frontend:** Vercel (Native support for Next.js, excellent free tier).

*   **Database:** Supabase (PostgreSQL with generous free tier).

## 6. Backend Models (SQLModel + Pydantic)

The backend must use SQLModel for ORM and Pydantic for validation.

Rules:

- SQLModel models must match Pydantic schemas
- No duplicated fields
- UUID primary keys
- Enums must be typed

Example:

```python
from sqlmodel import SQLModel, Field
from uuid import UUID, uuid4
from typing import Optional

class Recipe(SQLModel, table=True):
    id: UUID = Field(default_factory=uuid4, primary_key=True)
    title: str
    category: str
    yield_amount: float
    yield_unit: str

from pydantic import BaseModel
from uuid import UUID

class RecipeCreate(BaseModel):
    title: str
    category: str
    yield_amount: float
    yield_unit: str

```

## 7. API Contracts

All endpoints must return typed responses.

Rules:

- JSON only
- No any
- Must match Pydantic schema
- Must be used to generate TypeScript types

Example:

POST /api/recipes

Request:

```json
{
  "title": "Cake",
  "category": "Dessert"
}
```

Response:

```json
{
  "id": "uuid",
  "title": "Cake",
  "category": "Dessert"
}
```

## 8. Frontend Type Safety

Frontend must use TypeScript types generated from OpenAPI.

Rules:

- Do not use any
- Use typed fetch
- Use TanStack Query
- Use shadcn/ui components

Example:

type Recipe = {
  id: string
  title: string
  category: string
}

## 12. Frontend Design & UX (Culinary Zen)

O frontend do LaCuisine segue o conceito **Culinary Zen**, priorizando o minimalismo premium e a eficiência no ambiente de cozinha.

### 12.1 Princípios Básicos
- **Mobile-First:** Toda a interface é projetada primeiramente para dispositivos móveis.
- **AI-First:** A interação principal do sistema é através do Agente (Chat).
- **Interface Limpa:** Uso generoso de espaço negativo e tipografias elegantes.

### 12.2 Identidade Visual
- **Cores:**
    - Bege Claro (Creme): `#F4EFE7` (Fundo)
    - Preto Elegante (Grafite): `#111111` (Textos e Contornos)
    - Dourado Suave (Ouro): `#C9A46A` (Destaques e Botões)
- **Tipografia:**
    - Títulos: *Playfair Display*
    - Corpo/Interface: *Montserrat*

### 12.3 Estrutura de Telas
1. **Login/Registro:** Minimalista, apenas campos de dados sobre fundo sólido (sem imagens dispersoras).
2. **Dashboard (Home):** É a interface de chat com o Agente. Acesso a outras áreas via menu hamburguer.
3. **Livro de Receitas:** Grid de duas colunas. Cards expandem (via ícone de seta) para mostrar custos e rendimento sem sair da lista.
4. **Ficha Técnica:** Visual de tabela técnica com edição direta (in-place) para pesos e valores.

### 12.4 Navegação
- **Menu Hamburguer:** Navegação lateral para acesso ao Livro de Receitas, Menus e Perfil.
- **Barra de Ação de IA:** Fixa na base da Home (Chat) para inserção de texto ou fotos.
