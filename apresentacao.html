Filosofia de Design

Arquitetura Limpa nos Dois Lados

Adotamos a Clean Architecture de forma simétrica no Backend e no Frontend. O núcleo de negócios (Domínio) é 100% puro e isolado, garantindo testabilidade rápida e desacoplamento de infraestrutura.

  • Independência de Frameworks & DB As entidades e regras não dependem de FastAPI, Next.js, SQLAlchemy ou bibliotecas visuais. Se mudarmos o framework, o negócio permanece intacto.
  • Inversão de Dependência (D.I.) As camadas internas definem contratos (Ports) e as camadas externas fornecem implementações concretas (Adapters) que são injetadas.
  • Alta Testabilidade Como as entidades e casos de uso funcionam sem acesso à rede ou bancos, os testes unitários rodam isoladamente em frações de segundo.

Backend Engine

Backend: FastAPI & Inversão

O backend utiliza FastAPI para exposição de rotas de alto desempenho, SQLModel para unificar tipos Pydantic e modelos SQLAlchemy, e Alembic para versionamento de migrações estruturadas.

  • FastAPI Dependency Injection (Depends) Rotas utilizam injeção de dependência assíncrona. Injetamos o repositório concreto via dependência na rota e o passamos para os Use Cases.
  • Contratos Claros (Ports & Adapters) O domínio define RecipeRepositoryPort. A infraestrutura implementa RecipeRepository. O Use Case só depende da interface. 
# app/domain/entities/recipe.py
from pydantic import BaseModel
from app.domain.enums.recipe_enums import RecipeCategory

class RecipeBase(BaseModel):
    """Entidade base de receita no domínio (100% Pura)."""
    title: str
    category: RecipeCategory
    yield_amount: float
    yield_unit: str

class RecipeStepBase(BaseModel):
    """Entidade base de um passo da receita."""
    step_number: int
    instruction: str
# app/domain/ports/repository_ports.py
from abc import ABC, abstractmethod
from uuid import UUID

class RecipeRepositoryPort(ABC):
    """Interface (Porta) para o repositório de receitas."""

    @abstractmethod
    async def create(self, recipe_data: dict) -> dict:
        pass

    @abstractmethod
    async def get_by_id(self, recipe_id: UUID) -> dict | None:
        pass
# app/infrastructure/database/recipe_repository.py
from sqlalchemy.ext.asyncio import AsyncSession
from app.domain.ports.repository_ports import RecipeRepositoryPort
from app.infrastructure.database.models import RecipeModel

class RecipeRepository(RecipeRepositoryPort):
    """Implementação concreta (Adapter) usando SQLModel/SQLAlchemy."""

    def __init__(self, session: AsyncSession):
        self.session = session

    async def create(self, recipe_data: dict) -> RecipeModel:
        recipe = RecipeModel(**recipe_data)
        self.session.add(recipe)
        await self.session.commit()
        await self.session.refresh(recipe)
        return recipe
Metodologia TDD

Backend: TDD e Garantia de Qualidade

A testabilidade é a maior prova do sucesso de nossa arquitetura. Aplicamos TDD estrito escrevendo testes que falham antes do código de implementação, cobrindo três níveis da pirâmide de testes.

  • Testes Unitários de Negócio Validação de regras puras nas entidades (ex. validações Pydantic) e Use Cases usando mocks de repositório rápidos.
  • Integração com Rollback Automático Uso de banco SQLite assíncrono em memória (aiosqlite) com rollback automático após cada teste para garantir isolamento e velocidade.
  • Isolamento e Mocking de Serviços Mocking de serviços de rede (como a API do Gemini) usando AsyncMock e patching do Python unittest para testar fluxos sem gastar tokens.

Frontend Architecture

Frontend: Next.js & Estilo Moderno

O frontend é estruturado em Next.js 16 (App Router) com TypeScript. Focamos em performance na Edge, sincronização robusta de estado remoto via TanStack React Query e animações refinadas via Framer Motion.

  • Gestão de Cache e Estado Remoto Uso de TanStack React Query para persistência automática, cacheamento de receitas/menus e gerenciamento inteligente de mutações.
  • Estilização & Experiência Fluida Integração de Tailwind CSS v4 para designs modernos com transições suaves e Framer Motion para micro-interações elegantes.
  • Abstração de Rede com Axios Axios pré-configurado com intercepção global de erros e envio transparente de Cookies HTTP-Only para a segurança da sessão.

"use client";
import { useState } from "react";
import { QueryClient, QueryClientProvider } from "@tanstack/react-query";

export function Providers({ children }: { children: React.ReactNode }) {
  const [queryClient] = useState(() => new QueryClient({
    defaultOptions: {
      queries: {
        staleTime: 1000 * 60 * 5, // Cache por 5 minutos
        refetchOnWindowFocus: false,
      },
    },
  }));

  return (
    <QueryClientProvider client={queryClient}>
      {children}
    </QueryClientProvider>
  );
}
Frontend Clean & TDD

Frontend: TDD e Domínio Puro

A Clean Architecture no frontend é exemplificada pelo isolamento de regras de negócios. Nossas entidades e suas validações são escritas de forma agnóstica a frameworks (sem React) e cobertas por testes rápidos com Vitest.

  • Modelagem de Domínio Pura em TS Definição de tipos e funções de validação pura (ex. validateMenu) livres de dependências externas.
  • Vitest para Testes de Altíssima Velocidade Vitest substitui o Jest para execução instantânea dos testes unitários utilizando o compilador ultra-rápido do Vite.
  • React Testing Library para UI Validação de componentes com jsdom focando na acessibilidade e comportamento do usuário em vez de implementação interna. 
// frontend/src/domain/entities/Menu.ts
export interface MenuItem {
  category: string;
  recipeName: string;
}

export interface Menu {
  title: string;
  items: MenuItem[];
}

export function validateMenu(menu: Partial<Menu>) {
  const errors: Record<string, string> = {};
  if (!menu.title || menu.title.trim().length < 3) {
    errors.title = "O título do menu deve ter pelo menos 3 caracteres.";
  }
  if (!menu.items || menu.items.length === 0) {
    errors.items = "O menu deve ter pelo menos um prato listado.";
  }
  return errors;
}
// frontend/src/domain/entities/Menu.test.ts
import { describe, it, expect } from "vitest";
import { validateMenu } from "./Menu";

describe("Menu Entity Validation", () => {

Google Gemini Integration

Integração com IA (Google Gemini)

A plataforma se destaca no mercado pelo Copiloto Culinário com IA. Integramos o SDK moderno google-genai e o modelo gemini-2.0-flash para processar e analisar receitas através de imagens.

  • Extração Estruturada (Multimodal JSON) A IA recebe a foto de um prato ou cardápio e, através de prompts tipados, devolve uma estrutura JSON validada contendo título, ingredientes e passos.
  • Arquitetura de Serviços Desacoplada Definimos a interface AIServicePort no Domínio. A chamada à API de IA fica encapsulada na Infraestrutura sob o adaptador GeminiAIService. 
# app/infrastructure/ai/gemini_service.py
import json
from google import genai
from app.domain.ports.service_ports import AIServicePort

class GeminiAIService(AIServicePort):
    def __init__(self, api_key: str):
        self.api_key = api_key

    async def analyze_image(self, image_bytes: bytes) -> dict:
        client = genai.Client(api_key=self.api_key)
        
        # Chamada assíncrona multimodal
        response = client.models.generate_content(
            model="gemini-2.0-flash",
            contents=[
                {"inline_data": {"mime_type": "image/jpeg", "data": image_bytes}},
                "Extract recipe info and return strictly valid JSON..."
            ]
        )
        return json.loads(response.text.strip())
# app/domain/ports/service_ports.py
from abc import ABC, abstractmethod

class AIServicePort(ABC):
    """Contrato abstrato de serviço de IA."""
    
    @abstractmethod
    async def analyze_image(self, image_bytes: bytes) -> dict:
        """Processa a foto e retorna dicionário com os dados."""
        pass

    @abstractmethod
    async def suggest_menu(self, base_recipe: dict, available: list[dict]) -> dict:
        """Baseado nas receitas salvas, planeja um menu completo."""
        pass
DevOps & CI/CD

CI/CD: Automação GitHub Actions

A robustez operacional é assegurada por integração contínua (CI) via GitHub Actions. A cada commit ou pull request, o pipeline roda testes em ambos os lados e valida os builds dos containers Docker.

  • Duplo Teste Automatizado O pipeline executa testes de backend (Pytest com cobertura) e frontend (Vitest), bloqueando merges em caso de qualquer falha.
  • Docker Build Verification O workflow valida o build das imagens Docker do backend e frontend antes de liberar a entrega contínua.
  • Gatilho de CD para Render Quando os testes passam na branch main, um webhook seguro dispara o deploy imediato do backend no Render. 
# .github/workflows/ci.yml
name: CI/CD Pipeline
on:
  push:
    branches: [ main, master ]
  pull_request:
    branches: [ main, master ]

jobs:
  frontend-ci:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '20', cache: 'npm', cache-dependency-path: frontend/package-lock.json }
      - run: npm ci
        working-directory: ./frontend
      - run: npm run lint
        working-directory: ./frontend
      - run: npm run test
        working-directory: ./frontend

  backend-ci:
    runs-on: ubuntu-latest
    env:
      DATABASE_URL: "sqlite+aiosqlite:///:memory:"
      GEMINI_API_KEY: "dummy-key-for-testing"
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: { python-version: '3.11', cache: 'pip', cache-dependency-path: backend/pyproject.toml }
      - run: pip install -e ".[dev]"
        working-directory: ./backend
      - run: pytest --cov=app --cov-report=xml
        working-directory: ./backend

  deploy-backend:
    needs: [backend-ci, frontend-ci]
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    steps:
      - run: curl -f "${{ secrets.RENDER_DEPLOY_HOOK_URL }}"
Hospedagem & Deploy

Estratégia de Deploy Multicloud

A infraestrutura em nuvem do LaCuisine é distribuída estrategicamente para obter o menor custo, alta escalabilidade e deploy de latência zero para o cliente final.

  • Neon Postgres (Serverless DB) Autoscaling instantâneo e recurso de Scale-to-Zero (banco desliga quando inativo para economizar). Suporta branching lógico (cópia instantânea para testes).
  • Render Web Services (API FastAPI) Hospedagem em container Docker gerenciado. O build é otimizado via Dockerfile minimalista baseado em python:3.11-slim.
  • Vercel (Frontend Next.js) Deploy otimizado na Edge da Vercel. Cache global estático de páginas e reconstrução sob demanda (ISR), maximizando os Core Web Vitals.
